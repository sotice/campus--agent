# Campus Agent 测试用例

版本：V1.2  
关联文档：[SRS_软件需求规格说明书.md](./SRS_软件需求规格说明书.md)、[API接口设计文档.md](./API接口设计文档.md)  
测试范围：P0A/P0B Agent 闭环、Agent Run 暂停/恢复、核心演示链路、Tool Safety Gate、PendingAction、RAG 可信问答、失败 UX、Judge Mode / Debug Trace

## 1. 测试目标

本文档用于验证 Campus Agent 是否满足 MVP 需求，重点证明系统不是普通聊天机器人，而是具备完整事务闭环的校园 AI Agent。

核心目标：

1. 验证核心请求遵循 Agent Run 状态机和 `Observe → Plan → Act → Verify → Respond`。
2. 验证 Agent 能稳定生成结构化计划并选择 P0 工具。
3. 验证 Tool Safety Gate 能阻断非法工具、非法参数和未确认敏感操作。
4. 验证校园卡挂失等敏感操作必须经过 PendingAction。
5. 验证课表查询基于本地数据，可离线使用。
6. 验证校园知识问答基于知识库证据、来源和相关性分数，不编造。
7. 验证失败场景有明确 UX，不崩溃、不假成功、不泄露敏感信息。
8. 验证 Judge Mode / Debug Trace 可用于答辩展示，且完全脱敏、不可由 UI 伪造。
9. 验证 Android APK 至少能跑通一次核心饭卡闭环，作为移动端交付证据。

## 4. 测试优先级定义

| 优先级 | 定义 |
| --- | --- |
| P0 | 文档中的历史标记，默认等同 P0A；后续新增用例优先使用 P0A/P0B。 |
| P0A | 最小可获奖闭环必须通过，失败会影响项目验收。 |
| P0B | P0A 稳定后的冲刺增强，用于提升答辩观感和移动端证据。 |
| P1 | 重要增强或边界场景，应在正式演示前尽量通过。 |
| P2 | 后续扩展，可在比赛后完善。 |

| Unit Test | Agent Run 状态机 | `running → suspended_for_confirmation → resumed → completed`，取消和过期终态 | `flutter test` |
| Unit Test | RAG 策略 | 高相关回答、中相关带提示、低相关拒答 | `flutter test` |

| Manual Demo | 竞赛核心链路 | 饭卡闭环、课表查询、图书馆时间、安全失败路径 | Chrome Web + Android APK |

| TC-DEMO-002 | “那怎么补办？” | 多轮上下文、`knowledge.search`、来源卡 | 返回补办流程、source、updatedAt、score。 |
| TC-DEMO-003 | “我明天上午有课吗” | 本地工具调用、离线能力、Verify | 展示 `schedule.query` 和课程结果。 |
| TC-DEMO-004 | “图书馆今天几点关门” | RAG 来源、相关性分数、流式或状态回复 | 返回闭馆时间和来源卡，并说明 score 是相关性。 |
| TC-DEMO-005 | PendingAction 过期后确认 | 安全失败路径 | 不执行挂失，提示重新发起。 |
| TC-DEMO-006 | 低相关知识问题 | 幻觉防护 | 不编造，给人工/官方路径。 |
| TC-DEMO-007 | Debug Trace 检查 | 评委模式 | Trace 脱敏且展示 phase/tool/safety/evidence/duration。 |
| TC-DEMO-008 | Android APK 饭卡闭环 | 移动端交付证据 | Android 设备或模拟器完整跑通饭卡闭环，并保留截图或录屏。 |

## 10. 评委验收 Rubric

| 维度 | 得分点 | 扣分点 |
| --- | --- | --- |
| Agent 成色 | 有计划、有工具、有验证、有上下文 | 只有聊天文本或简单 intent 路由 |
| 安全闭环 | 未确认不执行，确认后参数冻结 | 模型可直接执行敏感操作 |
| 可信问答 | 来源、更新时间、trustLevel、score 相关性说明清晰 | 编造电话、地点、时间，或把 score 说成真实性概率 |
| 失败 UX | 过期、低相关、超时均有明确路径 | 无限 loading、假成功、崩溃 |
| 工程可行性 | Chrome Web + Built-in Mock 稳定复现 | 依赖现场后端/网络/真实学校接口 |
| 展示效果 | Judge Mode 中事务时间线、Safety Gate、PendingAction 倒计时和 Debug Trace 清楚 | 评委看不出与普通聊天机器人差别，或 UI 自行伪造阶段 |

## 11. 缺陷记录模板

| 字段 | 说明 |
| --- | --- |
| 缺陷 ID | 例如 BUG-20260707-001。 |
| 标题 | 简短描述问题。 |

## 12. 通过标准

MVP 可验收条件：

1. 所有 P0A/P0 用例通过。
2. P0B 尽量完成；P1 用例不阻塞 MVP，但不得影响核心演示。
3. 核心演示指令连续执行 3 次均成功。
4. 饭卡挂失未确认前不发生任何真实或模拟执行。
5. PendingAction 取消、过期、重复确认均安全处理。
6. RAG 低相关或无命中时不编造。
7. Debug Trace 无完整学号、完整卡号、Token、系统 Prompt、隐藏推理链，并且事件来自 Orchestrator。
8. Android APK 至少跑通一次饭卡闭环并留存截图或录屏。
9. 应用在核心流程中无崩溃、无明显卡死、无敏感信息明文日志。
