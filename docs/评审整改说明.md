# Campus Agent 评审整改说明

版本：V1.0  
整改依据：针对“概念伪智能、技术可行性、用户体验痛点、答辩亮点不足”的评审意见  
关联文档：[PRD.md](./PRD.md)、[SRS_软件需求规格说明书.md](./SRS_软件需求规格说明书.md)、[系统架构设计.md](./系统架构设计.md)、[AI_Agent设计文档.md](./AI_Agent设计文档.md)、[API接口设计文档.md](./API接口设计文档.md)、[测试用例.md](./测试用例.md)、[部署与运行说明.md](./部署与运行说明.md)

## 1. 整改目标

本次整改的目标不是继续堆功能，而是把 Campus Agent 从“可能被质疑为聊天机器人套壳”的方案，收敛为一个可证明、可实现、可演示的事务型 Agent MVP。

核心整改方向：

1. 用 Agent Run 状态机证明执行闭环，不让 UI 假装 Agent。
2. 用 PendingAction + `resume` 协议修复用户确认与流式事件之间的架构断点。
3. 用 `frozenParamsHash` 替代仅靠摘要校验，强化敏感操作安全边界。
4. 用 P0A/P0B 拆分交付范围，降低 8 月 30 日前延期风险。
5. 用等待阶梯、失败矩阵和低相关拒答降低等待焦虑与幻觉风险。
6. 用 Judge Mode 将技术亮点显性化，提升答辩辨识度。

## 2. 问题与整改处方

| 高频翻车点 | 原风险 | 整改动作 | 涉及文档 |
| --- | --- | --- | --- |
| 概念伪智能 | 时间线可能只是 UI 动画，不能证明真 Agent | 新增 Agent Run 状态机，Trace 必须由 Orchestrator 产生，UI 不得伪造阶段 | PRD、SRS、系统架构、AI Agent、测试用例 |
| 确认流程断点 | SSE/流式事件无法在同一条单向流里接收用户确认 | 新增 `suspended_for_confirmation` 与 `resume(runId, pendingActionId)` | SRS、系统架构、AI Agent、API |
| 敏感操作参数篡改 | `frozenParamsSummary` 只是展示摘要，不能做安全校验 | 新增 `frozenParamsCanonicalJson` 与 `frozenParamsHash`，Safety Gate 校验哈希 | SRS、AI Agent、API、测试用例 |
| 范围过大 | P0 同时包含过多能力，容易延期 | 拆分 P0A 最小可获奖闭环与 P0B 冲刺增强 | PRD、SRS、部署说明 |
| 大模型等待焦虑 | 只有“流式”要求，没有超时阶梯 | 增加 0-500ms、500ms-2s、2-8s、8-15s、15s+ 等待策略 | PRD、SRS |
| RAG 假精确 | `score=0.91` 容易被误解为真实性概率 | 明确 `score` 是检索相关性，不是真实性概率 | PRD、SRS、AI Agent、API、部署说明 |
| 答辩不惊艳 | 只有功能表，现场看不出技术差异 | 增加 Judge Mode：阶段节点、安全门、确认倒计时、Verify 状态变化、证据卡 | PRD、测试用例、部署说明 |
| 移动端属性不足 | Chrome Web 首选可能被质疑不是移动端项目 | 要求 Android APK 至少跑通一次校园卡闭环并保留截图或录屏 | PRD、SRS、测试用例、部署说明 |

## 3. 新的 MVP 策略

### 3.1 P0A：最小可获奖闭环

必须优先完成：

- 文本聊天与阶段状态。
- Agent Run 状态机。
- Tool Safety Gate。
- PendingAction + `resume`。
- `frozenParamsHash` 校验。
- Built-in Mock Service。
- 4 个核心工具：`schedule.query`、`campus_card.get_status`、`campus_card.report_loss`、`knowledge.search`。
- 校园卡丢失救援复合事务。
- 本地课表查询。
- 图书馆/补办流程知识问答。
- Judge Mode / Debug Trace。
- Android APK 闭环证据。

### 3.2 P0B：冲刺增强

P0A 稳定后再做：

- 大模型流式自然语言摘要。
- 低相关拒答安全失败 Demo。
- Judge Mode 微交互增强。

## 4. 必须避免的实现误区

1. 不要让 UI 自己点亮 Observe/Plan/Act/Verify/Respond；UI 只能渲染 Orchestrator 事件。
2. 不要把 `pending_action_created` 和敏感工具执行塞进同一条单向 SSE 流里假装闭环；用户确认只能通过 `resume(runId, pendingActionId)` 进入后半段。
3. 不要用 `frozenParamsSummary` 做安全校验，它只是用户可读摘要。
4. 不要把知识库 `score` 说成“答案可信度 91%”。
5. 不要一开始接真实校园 API、真实统一认证、服务端 RAG、完整课表 CRUD。
6. 不要只做 Chrome Web；至少准备 Android APK 的闭环截图或录屏。

## 5. 整改后验收重点

整改后，评委应能看到：

1. 输入“校园卡丢了，帮我挂失”后，Agent 创建 `runId`。
2. 系统查卡状态，Safety Gate 允许读操作。
3. 敏感挂失进入 `suspended_for_confirmation`。
4. 用户确认后触发 `resume(runId, pendingActionId)`。
5. Safety Gate 校验 `confirmationId` 与 `frozenParamsHash`。
6. 工具执行后 Verify 再次读取卡状态为 `lost`。
7. Agent 查询补办流程知识库并展示来源卡。
8. Judge Mode 展示 runId、sequence、phase、tool、safety、evidence、duration。
9. 低相关知识问题不会编造答案。
10. Android APK 至少跑通过同一条核心闭环。
