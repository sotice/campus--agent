# Campus Agent 部署与运行说明

版本：V1.1  
适用对象：开发者、测试人员、竞赛演示人员  
项目目录：`D:\Project\Flutter\campus_agent`

## 1. 项目概况

Campus Agent 是基于 Flutter 的事务型校园 AI Agent 应用。MVP 阶段的重点不是接入真实校园系统，而是稳定演示：

```text
Observe → Plan → Act → Verify → Respond
```

核心演示路径：

1. Chrome Web + Built-in Mock Service：首选，比赛现场最稳定。
2. Android APK + Built-in Mock Service：备用交付形态。
3. 本地 Mock API Server：可选工程展示，不作为 P0 依赖。

当前仓库为 Flutter 项目结构，核心配置在 `pubspec.yaml`，已包含 `flutter_ai_toolkit` 依赖。真实统一认证、真实校园 API、iOS、通知摘要、TTS 等均不是 MVP P0 交付项。

## 4. MVP 推荐运行方式

### 4.1 Chrome Web + Built-in Mock Service（首选）

适合快速验证 Agent 闭环、事务时间线、PendingAction、RAG 来源卡和 Debug Trace。此方式不需要启动后端服务。

```powershell
flutter run -d chrome --web-port 5173
```

启动成功后访问：

```text
http://127.0.0.1:5173
```

常用热更新命令：

| 命令 | 作用 |
| --- | --- |
| `r` | Hot reload。 |
| `R` | Hot restart。 |
| `q` | 退出运行。 |

### 4.2 Android APK + Built-in Mock Service（备用）

用于需要提交安装包或现场网络不稳定但有 Android 设备的情况。

查看设备：

```powershell
flutter devices
```

运行到 Android 设备：

```powershell
flutter run -d <device_id>
```

构建 APK：

```powershell
flutter build apk --release
```

构建产物通常位于：

```text
build\app\outputs\flutter-apk\app-release.apk
```

### 4.3 本地 Mock API Server（可选工程展示）

如果需要展示前后端分层和接口设计能力，可以启动本地 Mock API Server。注意：这不是 P0 必需路径，竞赛现场优先使用 Built-in Mock Service。

建议地址：

```text
http://127.0.0.1:8787/api/v1
```

真机调试时，手机不能访问电脑的 `127.0.0.1`，需要改成电脑局域网 IP：

```text
http://<电脑局域网IP>:8787/api/v1
```

Mock Server 至少支持：

- `POST /auth/demo-login`
- `GET /users/me`
- `POST /agent/pending-actions`
- `POST /agent/pending-actions/{pendingActionId}/confirm`
- `POST /agent/pending-actions/{pendingActionId}/cancel`
- `GET /campus-card/status`
- `POST /campus-card/report-loss`
- `POST /knowledge/search`

## 7. 核心 Agent 演示路径

### 7.1 Demo 1：饭卡丢失救援

输入：

```text
饭卡丢了，帮我挂失
```

必须观察到：

1. 事务时间线显示 Observe。
2. Plan 展示步骤：`campus_card.get_status` → PendingAction → `campus_card.report_loss` → Verify → `knowledge.search`。
3. Tool Safety Gate 对 `get_status` 返回 `allow`。
4. `campus_card.get_status` 返回 `normal` 和 `****0188`。
5. 对 `report_loss` 返回 `require_confirmation`。
6. UI 展示 PendingAction 确认卡。
7. 未点击确认时，卡状态不变。
8. 点击确认后签发 `confirmationId`，调用 `campus_card.report_loss`。
9. Verify 阶段再次确认 status 为 `lost`。
10. 最终回复展示挂失结果、补办建议、知识库来源。
11. Debug Trace 脱敏展示 phase、intent、tool、safety、evidence、duration。

追问：

```text
那怎么补办？
```

必须观察到：继承上一轮饭卡上下文，调用 `knowledge.search`，展示 `kb_card_replace_001` 来源。

### 7.2 Demo 2：本地课表查询

输入：

```text
我明天上午有课吗
```

必须观察到：

1. 调用 `schedule.query`。
2. 展示本地课表工具卡片。
3. Verify 阶段校验日期、上午、空结果或课程结果。
4. 关闭网络后仍可查询本地课表。

### 7.3 Demo 3：图书馆闭馆时间

输入：

```text
图书馆今天几点关门
```

必须观察到：

1. 调用 `knowledge.search`。
2. 返回 `kb_library_hours_001`。
3. 回复包含闭馆时间、source、updatedAt、trustLevel、score。
4. score 高于 0.80 时可直接回答。

### 7.4 Demo 4：安全失败路径

任选其一：

1. PendingAction 过期后点击确认：应提示确认已过期，未执行挂失。
2. 输入低置信知识问题：应提示未找到可靠来源，不编造具体答案。
3. 注入非法工具计划：Safety Gate 阻断，不执行工具。

6. `debugTraceEnabled=true`。
7. 演示账号 `demo_student_001` 可用。
8. 校园卡初始状态为 `normal`。
9. 预置课表数据存在。
10. 知识库包含图书馆开放时间和校园卡补办流程。

15. 核心 Demo 连续执行 3 次均成功。

## 9. 常见问题与处理

### 9.1 `flutter devices` 中 adb 启动失败

现象示例：

```text
cannot open C:\WINDOWS\TEMP\adb.log: Permission denied
failed to start daemon
```

处理方式：将当前 PowerShell 会话的临时目录改到用户目录后重试。

```powershell
$env:TEMP="$env:USERPROFILE\AppData\Local\Temp"
$env:TMP=$env:TEMP
flutter devices
```

若仍失败，检查 Android SDK、`ANDROID_HOME` 和 `platform-tools` 配置。

### 9.2 Chrome 端口被占用

如果 `5173` 被占用，可换端口：

```powershell
flutter run -d chrome --web-port 5174
```

### 9.3 Android 真机无法访问本地 Mock API

原因：手机上的 `127.0.0.1` 指向手机自身，不是电脑。

处理方式：

1. 确认手机和电脑在同一局域网。
1. 查看电脑 IP。

```powershell
ipconfig
```

1. 将 API 地址改为：

```text
http://<电脑局域网IP>:8787/api/v1
```

注意：MVP 推荐 Built-in Mock Service，因此普通演示不需要处理该问题。

### 9.4 Web 端接口跨域

仅使用本地 Mock API Server 时可能遇到 CORS 问题。Mock Server 需允许本地开发源：

```http
Access-Control-Allow-Origin: http://127.0.0.1:5173
Access-Control-Allow-Headers: Content-Type, Authorization, X-Request-Id, X-Demo-Mode
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
```

### 9.5 Flutter 依赖冲突

处理方式：

```powershell
flutter pub get
flutter pub outdated
```

若升级依赖，需重新运行核心 P0 测试，尤其是 Agent 闭环、PendingAction、Debug Trace。

## 11. 推荐开发流程

1. 阅读 [PRD.md](./PRD.md) 和 [SRS_软件需求规格说明书.md](./SRS_软件需求规格说明书.md)，确认收缩后的 P0 范围。
2. 先实现 Built-in Mock Service 和演示数据。
3. 实现 4 个 P0 工具：
   - `schedule.query`
   - `campus_card.get_status`
   - `campus_card.report_loss`
   - `knowledge.search`
4. 实现 Agent Orchestrator 的 `Observe → Plan → Act → Verify → Respond` 状态流。
5. 实现 Tool Safety Gate。
6. 实现 PendingAction Manager 和确认卡。
7. 实现事务时间线、工具调用卡、来源卡、Debug Trace。
8. 跑通饭卡丢失救援复合 Demo。
9. 跑通课表查询和图书馆时间 Demo。
10. 按 [测试用例.md](./测试用例.md) 执行所有 P0 回归。
11. 有余力再做语音输入、完整课表 CRUD、本地 Mock API Server 等 P1 能力。

## 12. 验收运行方式

竞赛现场推荐优先使用 Chrome Web。

运行命令：

```powershell
$env:TEMP="$env:USERPROFILE\AppData\Local\Temp"
$env:TMP=$env:TEMP
flutter pub get
flutter run -d chrome --web-port 5173
```

验收人员打开：

```text
http://127.0.0.1:5173
```

然后依次执行：

```text
饭卡丢了，帮我挂失
那怎么补办？
我明天上午有课吗
图书馆今天几点关门
```

再执行一个安全失败场景：

```text
PendingAction 过期后再确认
```

或：

```text
询问一个知识库低置信问题
```

通过标准：

1. 应用完成 Agent 闭环、状态展示、敏感确认、执行验证和结果反馈。
2. Tool Safety Gate 能证明未确认不执行。
3. RAG 来源卡清晰展示 source、updatedAt、trustLevel、score。
4. Debug Trace 清楚展示阶段、工具、安全决策、证据和耗时。
5. 全程无崩溃、无假成功、无敏感明文泄露。
